## 部署模型

Codex CLI 支持多种部署模型,以适应不同的用户需求和场景。

### 1. npm 包部署

**部署描述**:npm 包部署是最常用的部署方式,适合大多数开发者。

**部署特性**:

- **包名: `@openai/codex`**
- **入口: `codex.js` (Node.js)**
- **依赖:平台特定二进制包**
- **安装: `npm install -g @openai/codex`**

**部署流程**:

1. 安装 Node.js 环境(>= 16)
2. 执行 `npm install -g @openai/codex`
3. npm 自动下载平台特定的二进制包
4. 配置 PATH 环境变量
5. 验证安装成功

**部署优势**:

- 简单易用
- 自动平台检测
- 自动依赖管理
- 跨平台支持

**部署劣势**:

- 需要 Node.js 环境
- npm 依赖管理复杂
- 版本更新需要 npm 更新

### 2. Homebrew 部署

**部署描述**:Homebrew 部署适合 macOS 用户,提供原生应用体验。

**部署特性**:

- **类型:Cask (macOS 应用)**
- **位置:Applications 目录**
- **安装: `brew install --cask codex`**
- **更新: `brew upgrade codex`**

**部署流程**:

1. 安装 Homebrew
2. 执行 `brew install --cask codex`
3. Homebrew 下载 macOS 应用包
4. 安装到 Applications 目录
5. 验证安装成功

**部署优势**:

- macOS 原生应用
- 简单易用
- 自动更新管理
- 系统集成良好

**部署劣势**:

- 仅支持 macOS
- 需要 Homebrew 环境
- 应用包较大

### 3. 直接下载部署

**部署描述**:直接下载部署适合不想使用 npm 或 Homebrew 的用户。

**部署特性**:

- **来源:GitHub Releases**
- **格式:平台特定二进制**
- **平台:Windows/macOS/Linux**
- **安装:手动配置 PATH**

**部署流程**:

1. 从 GitHub Releases 下载二进制文件
2. 解压二进制文件
3. 手动配置 PATH 环境变量
4. 验证安装成功

**部署优势**:

- 不需要 Node.js 或 Homebrew
- 直接使用二进制文件
- 版本控制灵活
- 离线安装支持

**部署劣势**:

- 手动配置复杂
- 版本更新需要手动下载
- 平台检测需要手动处理

### 4. 开发环境部署

**部署描述**:开发环境部署适合开发者参与 Codex CLI 的开发和调试。

**部署特性**:

- **构建: `Cargo build`**
- **测试: `Cargo test`**
- **运行: `Cargo run`**
- **Bazel:可选构建系统**

**部署流程**:

1. 安装 Rust 和 Cargo
2. 克隆 Codex CLI 仓库
3. 执行 `Cargo build` 构建
4. 执行 `Cargo test` 测试
5. 执行 `Cargo run` 运行

**部署优势**:

- 开发和调试方便
- 版本控制灵活
- 自定义构建
- 测试支持

**部署劣势**:

- 需要 Rust 环境
- 构建时间较长
- 开发环境复杂

## 部署先决条件

### 1. macOS 先决条件

**系统要求**:

- macOS 10.15 或更高版本
- Intel 或 Apple Silicon 架构

**软件要求**:

- `/usr/bin/sandbox-exec` (Seatbelt)
- Node.js >= 16 (npm 安装)
- Git (版本控制)
- 网络访问 (API 调用)

**Seatbelt 要求**:

- macOS 原生沙箱机制
- 系统自带,无需额外安装
- 提供沙箱隔离能力

**Node.js 要求**:

- Node.js >= 16
- npm 包管理器
- 用于 npm 包部署

**Git 要求**:

- Git 版本控制
- 用于 Git 集成和代码管理
- 用于代码审查功能

**网络要求**:

- 网络连接
- 用于 API 调用
- 用于认证和更新

### 2. Linux 先决条件

**系统要求**:

- Linux 内核 5.13 或更高版本(Landlock 支持)
- x86_64 或 ARM64 架构

**软件要求**:

- bwrap (Bubblewrap)
- Landlock 支持
- Node.js >= 16 (npm 安装)
- Git (版本控制)

**Bubblewrap 要求**:

- bwrap 命令
- 用于 Linux 用户空间沙箱
- 提供进程隔离能力

**安装方式**:

- Debian/Ubuntu: `apt-get install bubblewrap`
- Fedora: `dnf install bubblewrap`
- Arch Linux: `pacman -S bubblewrap`

**Landlock 要求**:

- Linux 内核 5.13+
- 用于 Linux 内核沙箱
- 提供文件系统隔离能力

**检查方式**:

- 检查内核版本: `uname -r`
- 检查 Landlock 支持: `ls /sys/kernel/security/landlock`

**Node.js 要求**:

- Node.js >= 16
- npm 包管理器
- 用于 npm 包部署

**Git 要求**:

- Git 版本控制
- 用于 Git 集成和代码管理
- 用于代码审查功能

### 3. Windows 先决条件

**系统要求**:

- Windows 10 或更高版本
- x86_64 或 ARM64 架构

**软件要求**:

- Windows Sandbox (可选)
- Node.js >= 16 (npm 安装)
- Git (版本控制)
- PowerShell 5+

**Windows Sandbox 要求**:

- Windows Sandbox(可选)
- 用于 Windows 原生沙箱
- 提供沙箱隔离能力

**启用方式**:

- Windows 功能:启用 Windows Sandbox
- 需要 Windows 10 Pro/Enterprise

**Node.js 要求**:

- Node.js >= 16
- npm 包管理器
- 用于 npm 包部署

**Git 要求**:

- Git 版本控制
- 用于 Git 集成和代码管理
- 用于代码审查功能

**PowerShell 要求**:

- PowerShell 5+
- 用于命令执行
- 用于脚本支持

### 4. 通用先决条件

**API 要求**:

- OpenAI API 访问权限
- 用于 AI 模型调用
- 用于代码生成和审查

**获取方式**:

- 注册 OpenAI 账户
- 获取 API Key
- 配置认证信息

**网络要求**:

- 网络连接
- 用于 API 调用
- 用于认证和更新

**文件系统要求**:

- 文件系统访问
- 用于文件读写
- 用于会话存储

**终端要求**:

- 终端/命令行环境
- 用于 CLI 交互
- 用于 TUI 显示

## 部署工作流

### 阶段1:环境准备

**任务描述**:环境准备阶段负责检查和准备部署环境。

**主要任务**:

- 检查系统平台和架构
- 验证先决条件是否满足
- 准备安装目录和 PATH 配置

**平台检测**:

- 检测操作系统(Windows、macOS、Linux)
- 检测架构(x86_64、ARM64)
- 检测目标三元组

**先决条件验证**:

- 验证 Node.js 版本
- 验证 Git 安装
- 验证沙箱机制可用性
- 验证网络连接

**环境准备**:

- 准备安装目录
- 准备 PATH 配置
- 准备配置文件目录

### 阶段2:安装执行

**任务描述**:安装执行阶段负责执行安装过程。

**主要任务**:

- 选择安装方式(npm/Homebrew/直接下载)
- 下载平台特定二进制文件
- 配置环境变量和 PATH

**安装方式选择**:

- npm 包部署:适合大多数开发者
- Homebrew 部署:适合 macOS 用户
- 直接下载部署:适合不想使用包管理器的用户
- 开发环境部署:适合开发者

**二进制文件下载**:

- npm 自动下载平台特定二进制包
- Homebrew 下载 macOS 应用包
- 手动下载平台特定二进制文件

**环境配置**:

- 配置 PATH 环境变量
- 配置配置文件目录
- 配置认证信息目录

### 阶段3:认证配置

**任务描述**:认证配置阶段负责配置认证信息。

**主要任务**:

- 执行 `codex login`
- 选择认证方式(ChatGPT/API Key/设备代码)
- 存储认证信息到安全存储

**认证方式选择**:

- ChatGPT 登录:使用 ChatGPT 账户登录
- API Key:使用 OpenAI API Key
- 设备代码:使用设备代码认证

**认证流程**:

1. 执行 `codex login`
2. 选择认证方式
3. 输入认证信息
4. 验证认证有效性
5. 存储认证信息

**认证存储**:

- 存储到安全存储
- 存储到配置文件
- 存储到系统密钥链(macOS)

### 阶段4:配置管理

**任务描述**:配置管理阶段负责配置 Codex CLI。

**主要任务**:

- 创建 `~/.codex/config.toml`
- 配置沙箱策略、模型、MCP 服务器等
- 验证配置文件格式

**配置文件创建**:

- 创建 `~/.codex` 目录
- 创建 `config.toml` 文件
- 创建 `mcp.toml` 文件(可选)
- 创建 `hooks.toml` 文件(可选)

**配置内容**:

- 沙箱策略配置
- 模型配置
- MCP 服务器配置
- Hook 配置
- 网络策略配置

**配置验证**:

- 验证配置文件格式(TOML)
- 验证配置值合法性
- 验证配置完整性

### 阶段5:功能验证

**任务描述**:功能验证阶段负责验证 Codex CLI 功能。

**主要任务**:

- 执行 `codex doctor` 诊断
- 测试基本功能(代码生成、执行)
- 验证沙箱机制和网络策略

**诊断检查**:

- 检查系统状态
- 检查配置状态
- 检查认证状态
- 检查沙箱状态
- 检查网络状态

**功能测试**:

- 测试代码生成功能
- 测试代码执行功能
- 测试文件操作功能
- 测试会话管理功能

**沙箱验证**:

- 验证沙箱机制可用性
- 验证沙箱隔离能力
- 验证沙箱权限控制

**网络验证**:

- 验证网络策略配置
- 验证网络访问控制
- 验证网络连接

### 阶段6:生产使用

**任务描述**:生产使用阶段负责开始使用 Codex CLI。

**主要任务**:

- 配置扩展(MCP 服务器、插件、技能)

**启动方式**:

**扩展配置**:

- 配置 MCP 服务器

**更新和维护**:

## 部署架构

### 用户环境

**环境组成**:

- 终端/命令行:用于 CLI 交互
- Node.js 环境:用于 npm 包部署
- Git 仓库:用于 Git 集成

**环境配置**:

**CLI 组成**:

- CLI 入口: `codex.js` (Node.js)

- TUI 界面: `codex-rs/tui` (Ratatui)
- 核心业务: `codex-rs/core` (Rust)
- 协议层: `codex-rs/protocol` (Rust)
- 基础设施: `codex-rs/sandboxing, config, exec, state` (Rust)

**CLI 功能**:

- 命令解析和分发
- TUI 界面渲染
- 核心业务处理
- 协议和 API 调用
- 基础设施支持

### 外部服务

**服务组成**:

- OpenAI API:AI 模型调用
- 认证服务:认证管理
- MCP 服务器:工具扩展
- 应用服务器:守护进程和远程控制
- WebSocket 服务:实时通信

**服务调用**:

- AI 模型调用:调用 GPT-4 或 Codex
- 认证调用:认证验证和刷新
- MCP 调用:调用 MCP 工具
- 应用服务器调用:远程控制
- WebSocket 调用:实时通信

### 数据存储

**存储组成**:

- SQLite 数据库:会话和状态存储
- 文件系统:配置和认证存储
- Rollout 记录:执行过程记录
- 缓存数据:Context 和历史缓存

**存储管理**:

- 数据持久化:保存会话和状态
- 数据恢复:恢复会话和状态
- 数据清理:清理旧数据
- 数据备份:备份重要数据
